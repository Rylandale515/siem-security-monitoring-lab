# Incident Analysis Report
**Capstone Project – SIEM Monitoring Lab**

---

# 1. Incident Details

At **19:30 UTC on February 22, 2026**, two endpoint systems within the organization’s network environment were compromised. The incident occurred at the organization’s primary office located in **Tahlequah, Oklahoma**.

The attackers gained unauthorized access through a **brute-force password attack** targeting a Windows workstation. After successfully guessing valid user credentials, the attackers authenticated to the workstation and established access.

Following the initial compromise, the attackers performed several actions to escalate privileges and maintain persistence within the environment. These actions included:

- Creating a **new Active Directory (AD) user account**
- Adding the newly created account to the **Administrators group**
- Modifying host firewall rules
- Creating **scheduled tasks**
- Changing permissions on classified files
- Modifying system security configurations
- Starting and stopping system services

The attack was **contained to two endpoint machines** and did not propagate to the Wazuh SIEM server. However, the SIEM successfully recorded multiple indicators of compromise throughout the attack lifecycle.

The activity is considered **malicious**, as the attackers compromised system integrity and established persistence mechanisms through administrative privilege escalation.

# 4. Initial Detection and Alerting

The incident was detected through **alerts generated by the Wazuh SIEM platform**.

Security logs collected from the Windows endpoint triggered alerts indicating:

- Multiple failed authentication attempts
- Successful authentication following repeated failures
- Creation of a new administrative account
- Security policy modifications

A **SOC analyst** reviewing the Wazuh dashboard identified suspicious behavior and escalated the alert at **19:46 UTC** on February 22, 2026.

---

# 5. Attack Vector and Techniques

The attackers gained access via a **password brute-force attack** targeting a Windows workstation.

This vulnerability existed due to:

- Username persistence from previous login sessions
- Lack of account lockout policy enforcement

Once access was obtained, the attackers executed several post-compromise techniques.

### Observed Techniques

| Technique | Description |
|----------|-------------|
| Credential Brute Force | Repeated password attempts to gain authentication |
| Privilege Escalation | Creation of an administrative account |
| Persistence | Administrator account maintained system access |
| Defense Evasion | Firewall rule modification |
| System Manipulation | Scheduled tasks and service configuration changes |

### MITRE ATT&CK Mapping

| Technique ID | Technique |
|--------------|-----------|
| T1110 | Brute Force |
| T1136 | Create Account |
| T1053 | Scheduled Task |
| T1562 | Impair Defenses |

# 7. Containment Actions

Once the incident was confirmed, the following containment actions were taken:

1. Both compromised endpoint systems were **isolated from the network**
2. Systems were **physically disconnected from network access**
3. Administrative credentials associated with the compromised accounts were disabled
4. Network monitoring was increased to detect additional suspicious activity

These measures prevented the attack from spreading further within the environment.

---

# 8. Eradication Efforts

Following containment, eradication procedures were performed.

Actions included:

- Removal of unauthorized administrative accounts
- Restoring systems from **verified clean backups**
- Reapplying secure system configurations
- Implementing **account lockout policies**
- Enforcing login procedures requiring manual username entry
- Implementing **least-privilege access controls**

These actions removed persistence mechanisms introduced by the attackers.

---

# 9. Recovery Steps

After eradication was completed, system recovery procedures were executed.

The affected systems were:

- Restored from a backup state taken prior to the compromise
- Verified for system integrity
- Tested to confirm operational functionality

Security policies were deployed across all endpoints to prevent similar incidents in the future.

# 13. Lessons Learned

The incident revealed several weaknesses in the organization's security posture.

Key lessons include:

- Weak password policies increase brute-force attack risk
- Login session persistence exposes usernames
- Lack of account lockout policies increases attack success probability
- SIEM monitoring significantly improves incident detection

---

# 14. Recommendations and Action Items

To prevent similar incidents in the future, the following improvements are recommended:

1. Implement **account lockout policies**
2. Enforce **strong password requirements**
3. Deploy **multi-factor authentication (MFA)**
4. Conduct regular **security awareness training**
5. Maintain **continuous SIEM monitoring**
6. Perform periodic **incident response simulations**

These measures will strengthen the organization’s security posture and improve resilience against unauthorized access attempts.
